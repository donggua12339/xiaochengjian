# ADR 0002 · 部署形态与商业模式

- 状态:accepted
- 日期:2026-07-13
- 决策者:小城笺项目
- 层次:战略

## 背景

卡密验证系统的部署形态决定数据库结构、用户体系、计费系统、服务器架构、安全边界。参考的 3 个开源项目(orange-resource/verify、byst4nder/netauthos、GuYiovo/GuYi-Access)均为开源,但 Arm Pro/小纸片等商业 SaaS 验证了订阅模式可行。

## 决策

### 部署形态:开源 + SaaS 双模式
- **开源版**:功能精简,单租户,核心验证逻辑完整,自部署
- **SaaS 版**:多租户、运维面板、SLA 监控、高级功能、托管运维

**SaaS 独有功能**:
- 多租户隔离与计费
- 安卓端注入工具(上传 APK 到服务器处理)
- 安卓端管理 APP
- 高级监控告警
- SLA 保障

**开源版独有**:
- 单租户核心验证
- 注入工具 CLI(本地运行)
- 完整文档

### 商业模式:开发者订阅
- **月度会员**:按月付费,基础功能
- **终身会员**:一次性付费,永久使用
- **VIP 解锁**:在月度/终身基础上,解锁高级功能(如多应用、高级统计、API 高调用上限)

不采用按调用量抽成(开发者抵触)、按卡密数量抽成(账务复杂)。

## 备选方案

| 方案 | 优点 | 缺点 | 不选原因 |
|---|---|---|---|
| (a) SaaS 多租户 | 接入成本低、可订阅 | 承担运维与攻击风险 | 缺少社区引流 |
| (b) 开源自部署 | 零运维、社区贡献 | 无法商业化 | 无收入支撑长期 |
| (c) 开源 + SaaS 双模式(本方案) | 兼顾社区与商业 | 双线维护 | 可接受 |
| (d) 商业私有化部署 | 单笔收入高 | 销售成本高 | 目标客户窄 |

## 影响

- 正面:社区引流 + 商业变现双轨
- 负面:SaaS 独有功能需用 feature flag 控制,编译时排除
- 风险:开源版被竞争者拿去做 SaaS,需用 Apache 2.0 + 商业版差异化应对
