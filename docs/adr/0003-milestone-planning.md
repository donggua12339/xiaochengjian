# ADR 0003 · 里程碑规划

- 状态:accepted
- 日期:2026-07-13
- 决策者:小城笺项目
- 层次:战略

## 背景

小城笺 MVP 范围广(后端 + 后台 + SDK + 注入工具 + 部署 + 安卓管理 APP),单人开发需要清晰阶段划分避免范围蔓延。

## 决策

### 4 里程碑

| 里程碑 | 范围 | 验收标准 |
|---|---|---|
| M0 基建 | Monorepo 初始化、CI 流水线、文档骨架、ADR 全部写入、CLA 模板 | 空 PR 能跑 CI,文档可读 |
| M1 后端 + 后台 | NestJS 核心 API + Vue 后台 + 多租户 + 2FA + 限流 | 开发者能注册、创建应用、生成卡密、查看统计 |
| M2 SDK + Rust 核心 | Rust so 核心安全模块 + Kotlin SDK + 反调试 + 离线缓存 | demo APP 集成 SDK,卡密激活验证全流程通 |
| M3 注入 + 部署 | 注入工具(含安卓端 APP) + Compose 部署 + 监控 + 安卓管理 APP | 端到端:注入 APK -> 安装 -> 激活 -> 验证 |

### MVP 不做的功能(明确写进 README "Not in MVP")
- 价值卡/储值卡
- 转赠/退卡/续费
- 卡密动态生成(对接支付)
- PC SDK、Unity 插件
- 加固 APK 自动脱壳(放 v1.1)
- 数据大屏、渠道分销
- K8s 多区域、LTS 版本
- 自动部署到 SaaS 生产
- E2E 测试自动化

### 每 milestone 结束做安全 review
对照 ADR 0019-0027 安全决策查漏补缺。

## 备选方案

| 方案 | 优点 | 缺点 | 不选原因 |
|---|---|---|---|
| 顺序 M0->M3(本方案) | 阶段清晰、依赖明确 | 工期长 | 合理 |
| 并行多 milestone | 快 | 单人无法并行 | 不可行 |
| 先 SDK 后后端 | 优先验证客户端 | 后端未就绪无法测试 | 依赖颠倒 |

## 影响

- 正面:范围明确,避免蔓延
- 负面:M1 完成前 SDK 无法测试,需 mock 后端
- 风险:M2 Rust 核心是技术难点,可能延期
