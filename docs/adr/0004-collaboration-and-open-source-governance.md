# ADR 0004 · 协作模式与开源治理

- 状态:accepted
- 日期:2026-07-13
- 决策者:小城笺项目
- 层次:战略

## 背景

开源 + SaaS 双模式下,需要明确代码贡献、分支策略、CLA、安全模块治理。

## 决策

### 协作模式:单人开发 + 开源接收贡献
- MVP 阶段项目作者单人推进
- 开源后接收社区贡献,但**核心 Rust 安全模块不接受外部 PR**,只接受 issue
- 贡献者必须签署 CLA

### 分支策略
- `main`:受保护,只接收 PR 合并,对应生产
- `dev`:开发分支,功能集成
- `feat/xxx`、`fix/xxx`、`docs/xxx`:功能分支
- 标签:`v1.0.0` 语义化版本

### PR 流程
- PR 必须含变更说明 + 影响范围 + 测试方式
- CI 必须 lint + 单测通过
- 单人开发阶段项目作者自己 review,流程要走

### CLA(贡献者协议)
- 个人 CLA + 企业 CLA 两种
- 签署后才能合并 PR
- 模板见 `docs/cla.md`

### 开源治理
- License:Apache 2.0
- issue 模板:bug / feature / security 三种
- PR 模板:含变更说明 + 影响范围 + 测试方式
- 行为准则:Contributor Covenant

### SaaS 独有功能隔离
- 开源版与 SaaS 版**同一仓库**
- SaaS 独有功能用 feature flag(`ENABLE_SAAS` 编译时变量)控制
- 编译开源版时排除 SaaS 模块,避免代码泄露

## 备选方案

| 方案 | 优点 | 缺点 | 不选原因 |
|---|---|---|---|
| 单人 + 开源(本方案) | 决策快、社区引流 | 单点风险 | 合理 |
| 小团队 | 并行开发 | 招人成本高 | MVP 阶段不需要 |
| 完全闭源 | 商业保护 | 无社区 | 与双模式矛盾 |

## 影响

- 正面:开源引流,CLA 保护商业授权
- 负面:单人开发瓶颈,核心模块不接受 PR 会让部分贡献者失望
- 风险:CLA 流程可能吓退贡献者,需简化电子签署
