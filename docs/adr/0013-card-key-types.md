# ADR 0013 · 卡密类型与生命周期

- 状态:accepted
- 日期:2026-07-13
- 决策者:小城笺项目
- 层次:功能

## 背景

卡密验证系统需要支持多种卡密类型,但 MVP 范围需精简。同时需明确卡密生命周期操作。

## 决策

### MVP 卡密类型
- **时间卡**:日卡 / 周卡 / 月卡 / 永久卡
- **试用卡**:用户主动领取(设备级判重)

### 不做的类型
- 次数卡(放 v2)
- 价值卡/储值卡(放 v2,涉及实时扣费账务)
- 会员卡(与开发者订阅模式概念冲突)
- 年卡(MVP 不做,放 v2)

### 机器码绑定作为卡密属性
- 机器码绑定**不是独立卡密类型**,而是卡密的**属性**
- 一张月卡可配置"绑定/不绑定",而非"月卡"vs"机器码绑定月卡"两种类型
- schema 更干净

### 试用卡机制
- **设备级领取**:基于机器码判重,一台设备只能领一次
- **无最终用户账号系统**:验证系统只认设备指纹,用户系统是开发者职责
- 试用卡本质 = 未激活的卡密 + 设备级判重

### 卡密生命周期(MVP)
| 操作 | MVP | v2 |
|---|---|---|
| 激活 | ✅ | |
| 验证 | ✅ | |
| 禁用 | ✅ | |
| 解绑 | ✅ | |
| 转赠 | ❌ | ✅ |
| 退卡 | ❌ | ❌ |
| 续费 | ❌ | ❌ |

### 不做的操作理由
- **转赠**:与"首次激活绑定"逻辑冲突,开发者通常用分销系统解决
- **退卡**:与开发者收入逻辑冲突,涉及退款账务
- **续费**:简化为"旧卡到期后用新卡激活"

## 备选方案

| 方案 | 优点 | 缺点 | 不选原因 |
|---|---|---|---|
| 全类型支持 | 功能完整 | MVP 工期翻倍 | 范围蔓延 |
| 仅时间卡 | 最简 | 缺试用 | 试用是开发者刚需 |
| 时间卡 + 试用卡(本方案) | 平衡 | - | 合理 |

## 影响

- 正面:schema 简洁,MVP 工期可控
- 负面:次数卡用户需等 v2
- 风险:试用卡被滥用刷设备,需配合 D3 防爆破
