# ADR 0014 · 卡密格式与生成策略

- 状态:accepted
- 日期:2026-07-13
- 决策者:小城笺项目
- 层次:功能

## 背景

卡密格式影响用户体验(输入)、安全性(防枚举)、存储设计(明文 vs hash)。生成策略影响开发者工作流。

## 决策

### 卡密格式:分段式 4x4 字母数字 + Luhn 校验位
- **格式**:`XXXX-XXXX-XXXX-XXXX`(16 字符,4 段)
- **字符集**:32 个字母数字(去掉易混淆字符 `0/O/1/I/L`)
- **校验位**:最后 1 位 Luhn 校验(防输错)
- **大小写**:统一大写,输入时小写自动转大写
- **熵**:31 × log2(32) = 155 bit,远超暴力枚举阈值

### 示例
```
A3K9-X2M7-Q1P5-R8T2
```

### 服务端存储
- **不存明文**:服务端只存 `SHA-256(cardKey + perCardSalt)`
- **校验时**:用户输入 -> 计算 hash -> 对比数据库 hash
- **perCardSalt**:每张卡密独立盐,防彩虹表

### 生成策略
| 策略 | MVP | v2 |
|---|---|---|
| (a) 批量生成 | ✅ | |
| (b) 模板化生成 | ✅ | |
| (c) 动态生成(对接支付) | ❌ | ❌ |
| (d) 导入外部卡密 | ❌ | ✅ |

### MVP 生成工作流
1. 开发者后台选择应用 + 卡密类型(日/周/月/永久/试用) + 数量 + 绑定策略
2. 后端批量生成,服务端只存 hash
3. 后端返回**明文卡密列表**(此时期是开发者唯一拿到明文的机会)
4. 开发者下载 CSV 导出
5. 服务端**不保留明文**,后续无法再次导出

### 模板化生成
- 预设套餐模板(如"月卡批量 1000 张")
- 模板可保存复用
- 本质是批量生成的快捷方式

## 备选方案

| 格式 | 优点 | 缺点 | 不选原因 |
|---|---|---|---|
| 纯数字 16 位 | 易输入 | 易枚举 | 不安全 |
| 字母数字 12 位 | 防枚举 | 易错输 | 体验差 |
| 分段 4x4(本方案) | 易读防错 | 占空间 | 合理 |
| 分段 + Luhn(本方案) | 防错输 | 1 位损失熵 | 合理 |

## 影响

- 正面:用户体验好,防错输,防枚举
- 负面:开发者导出 CSV 后丢失即无法找回
- 风险:Luhn 算法公开,攻击者可生成合法格式但无法通过服务端 hash 校验
