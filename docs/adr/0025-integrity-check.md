# ADR 0025 · 完整性校验与防重打包

- 状态:accepted
- 日期:2026-07-13
- 决策者:小城笺项目
- 层次:安全

## 背景

开源 + Smali 重打包方案下,攻击者可重打包 SDK 注入恶意逻辑。需完整性校验。

## 决策

### 完整性校验组合

| 方案 | 实现 |
|---|---|
| (a) APK 签名校验 | Rust so 内置预期签名 hash,运行时获取 APK 签名对比 |
| (c) so 自校验 | Rust so 启动时 hash 自己的 .text 段,对比编译时嵌入值 |
| (d) 服务端下发校验值 | 校验值不硬编码,每次启动从服务端拉取对比 |

### 不做 dex 完整性校验
- dex 在开源场景下本就公开,校验无意义

### 签名校验实现
1. Rust so 内置预期签名 hash
2. 运行时通过 `PackageManager.GET_SIGNATURES` 获取 APK 签名
3. **同时直接读 APK 文件解析签名**(不依赖系统 API,防 hook)
4. 两种方式获取的签名都对比预期 hash
5. 任一不匹配 -> 标记异常,延迟拒绝

### so 自校验实现
1. 编译时计算 .text 段 hash,嵌入数据段
2. 启动时读取 .text 段,计算 hash
3. 对比嵌入值
4. 不匹配 -> 标记异常

### 服务端下发校验值
1. 开发者后台配置"预期签名 hash 列表"(可多个)
2. 原始 APK 签名 + 注入后重签的签名都加入
3. SDK 启动时拉取对比
4. 攻击者改 SDK 后无法同步更新服务端配置

### Smali 重打包兼容
- Smali 重打包注入工具会改变 APK 签名
- 开发者后台配置"允许的签名 hash 列表"(可多个)
- 原始 APK 签名 + 注入后重签的签名都加入白名单

## 备选方案

| 方案 | 强度 | 不选原因 |
|---|---|---|
| 仅签名校验 | 中 | 易 hook 系统 API |
| 签名 + so 自校验 | 高 | 无服务端动态 |
| 签名 + so 自校验 + 服务端下发(本方案) | 最高 | 合理 |

## 影响

- 正面:防重打包,防 so 被 patch
- 负面:开发者换签名密钥需同步更新后台配置
- 风险:服务端下发校验值缓存被篡改,需结合 HMAC 签名
