# ADR 0029 · 加固 APK 兼容性

- 状态:accepted
- 日期:2026-07-13
- 决策者:小城笺项目
- 层次:安全

## 背景

加固 APK(360/腾讯乐固/爱加密/梆梆)的 dex 是壳,直接 dex 操作会失败。需明确兼容策略。

## 决策

### 加固 APK 兼容策略:壳 dex 默认 + 自动脱壳可选

| 模式 | 说明 | 默认 |
|---|---|---|
| 壳 dex 注入 | 在壳 dex 里插入加载器,运行时 hook 业务 dex | ✅ 默认 |
| 自动脱壳后注入 | 集成 FRIDA-DEXDump 脱壳后注入真实 dex | 可选(需免责协议) |

### ⚠️ 法律风险声明
**自动脱壳涉及绕过技术保护措施,可能违反:**
- 《中华人民共和国著作权法》第 48 条
- 《计算机软件保护条例》第 24 条

### 风险承担
- 项目作者明确愿意承担法律风险
- 用户协议中明确"用户保证拥有 APK 著作权或合法授权,承担一切法律后果"
- 项目保留配合司法调查的权利
- 项目 README 明确"自动脱壳功能仅限合法授权场景使用"

### 免责协议签署流程
1. 开发者首次使用脱壳功能前强制签电子协议
2. 协议明确:用户保证拥有 APK 合法授权
3. 协议明确:用户承担一切法律后果
4. 协议明确:平台保留配合司法调查权利
5. 签署记录审计日志

### 壳 dex 注入实现
- 检测 APK 加固特征(特定 so 文件、特定 dex 结构)
- 在壳 dex 里插入 `XiaochengjianSDK.init(this)` 调用
- 运行时通过 hook(Java 反射 / native hook)接管业务 dex 启动
- 兼容性中等:部分加固方案会阻止 hook

### 自动脱壳实现
- 集成 FRIDA-DEXDump 或类似工具
- 脱壳后注入真实 dex
- 脱壳过程记录审计日志
- 脱壳产物不持久化,注入后立即删除

### 检测加固
- 扫描 APK 是否有 360/腾讯/爱加密/梆梆等加固特征
- 检测到加固时提示开发者选择"壳 dex 注入"或"自动脱壳"
- 不强制拒绝加固 APK(用户明确选择)

## 备选方案

| 方案 | 优点 | 缺点 | 不选原因 |
|---|---|---|---|
| 不支持加固 | 零法律风险 | 加固 APK 用户流失 | 用户选择承担风险 |
| 仅壳 dex 注入 | 法律风险低 | 兼容性中 | 用户选择脱壳 |
| 仅自动脱壳 | 兼容性高 | 法律风险高 | 缺壳 dex 降级 |
| 壳 dex 默认 + 脱壳可选(本方案) | 灵活 | 法律风险 | 用户选择 |

## 影响

- 正面:覆盖加固 APK 场景,提升兼容性
- 负面:法律风险显著,需严格免责协议
- 风险:被加固厂商起诉,需保留法律应对预案
