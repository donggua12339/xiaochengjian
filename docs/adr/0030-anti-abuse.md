# ADR 0030 · 防滥用机制

- 状态:accepted
- 日期:2026-07-13
- 决策者:小城笺项目
- 层次:安全

## 背景

开源 + SaaS 双模式下,注入工具会公开。攻击者可拿注入工具给任意 APK 注入"小城笺验证"卖钱,等于给灰产提供工具。

## 决策

### 防滥用组合

| 机制 | MVP | v2 |
|---|---|---|
| (a) 签名密钥强制要求 | ✅ | |
| (c) 注入水印 | ✅ | |
| (b) 包名白名单 | ❌ 故意去掉 | - |
| (d) 在线激活 | ❌ 开源版必须离线可用 | - |
| (e) 审计日志 | SaaS 版自然有 | - |

### (a) 签名密钥强制
- 注入工具启动时**强制要求用户提供 keystore 文件 + 密码**
- 不提供则拒绝注入
- 攻击者必须有自己的签名才能注入

### (c) 注入水印
- 注入后的 APK 在 metadata 中写入:
  - 小城笺注入版本
  - 开发者 ID(SaaS 版)
  - 时间戳
  - 随机 nonce
- 水印**加密存储**(AES-256,密钥服务端持有)
- 攻击者难以擦除水印
- 被滥用时可追溯

### 不做 (b) 包名白名单
- 用户明确选择去掉,降低维护成本
- 包名白名单能被改包名绕过
- 法律追溯依靠水印即可

### 不做 (d) 在线激活
- 开源版必须离线可用,与开源自部署矛盾
- SaaS 版注入工具自然在线

### 不做 (e) 审计日志(开源版)
- 开源版无服务端,无法上报
- SaaS 版注入走服务器自然有日志

### 法律层面
- 用户协议明确"禁止用于注入他人 APK"
- 发现滥用可封号
- 注入水印提供追溯依据

### 重打包签名方案
- **V1 + V2 + V3**(apksigner)
- V1 兼容老设备
- V2 是 Android 7+ 必需
- V3 支持密钥轮换
- 不做 V4(增量安装,与注入无关)

## 备选方案

| 方案 | 强度 | 不选原因 |
|---|---|---|
| 仅签名强制 | 低 | 无追溯 |
| 签名 + 水印(本方案) | 中 | 用户选择 |
| 签名 + 水印 + 包名白名单 | 高 | 维护成本高 |
| 签名 + 水印 + 在线激活 | 最高 | 开源版不可用 |

## 影响

- 正面:基础防滥用 + 追溯能力
- 负面:无包名白名单,攻击者可用自己签名+自己包名绕过
- 风险:开源工具的根本限制,只能靠法律约束 + 追溯水印
