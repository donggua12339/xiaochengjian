# ADR 0031 · SaaS 部署架构

- 状态:accepted
- 日期:2026-07-13
- 决策者:小城笺项目
- 层次:运维

## 背景

SaaS 版需支撑多租户、高并发验证请求。MVP 阶段用户少,但架构需为升级留口子。

## 决策

### SaaS 部署架构(分阶段)

| 阶段 | 架构 |
|---|---|
| (a) MVP 单机 | 1 台 4C8G:Docker Compose 跑 NestJS + PG + Redis + Nginx |
| (b) 小规模 | 2 台:应用服务器 + DB 服务器,PG 主从 |
| (c) 中规模 | K8s 集群:3 节点 + PG 主从 + Redis 哨兵 |
| (d) 大规模 | K8s 多区域 + PG 分片 + Redis 集群 + CDN |

### MVP 阶段:(a) 单机
- 1 台 4C8G 云服务器
- Docker Compose 跑全部服务
- 月成本 < 200 元
- QPS < 1000 无压力

### 预设升级路径
- PG 配置文件预写主从
- Redis 配置预写哨兵模式
- NestJS **无状态化**(不存本地 session,所有状态走 Redis)
- 所有配置走环境变量,单机/集群切换只改 `.env`

### 不上 K8s 的时机
- QPS < 500
- 开发者数 < 1000
- 单机扛得住就不上,降低运维复杂度

### 多区域容灾
- 放 v2,MVP 单可用区
- 大规模阶段(d)考虑多区域

### 数据库主从
- 小规模阶段(b)启用 PG 主从
- 主写从读,验证查询走从库
- WAL 流复制,RPO 5 分钟

### Redis 哨兵
- 中规模阶段(c)启用
- 自动故障转移
- 配合 PG 主从

## 备选方案

| 方案 | 优点 | 缺点 | 不选原因 |
|---|---|---|---|
| 一开始就 K8s | 强大 | 运维复杂 | MVP 过度设计 |
| 单机起步 + 预设升级(本方案) | 务实 | 需重构 | 合理 |
| Serverless | 弹性 | 冷启动 | 长连接场景差 |

## 影响

- 正面:MVP 成本低,升级路径清晰
- 负面:单机阶段单点故障,需快速响应
- 风险:升级时重构成本,需预留接口
