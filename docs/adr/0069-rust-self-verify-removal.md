# ADR 0069 · Rust so 自校验撤除:服务端验证权威

- 状态:accepted
- 日期:2026-07-19
- 决策者:小城笺项目
- 层次:安全

## 背景

ADR 0058 决策"Rust so 自校验:编译时嵌入 .text 段 hash + 服务端下发预期 hash 组合",要求:
1. `build.rs` 读取 .text 段计算 hash,嵌入常量(obfstr 混淆)
2. JNI 入口(native01-08)各自校验不同段
3. 服务端开发者后台配"预期 hash",SDK 启动拉取对比

2026-07-15 ~ 2026-07-18 的 v2 重构期间,前任工程师撤掉了 so 自校验设施。代码事实:

- `sdk-android/rust/xcj-core/` 目录无 `build.rs` 文件
- `sdk-android/rust/xcj-core/src/integrity.rs` 顶部注释明确写:"v2 已撤掉 so 自校验(.text 段 hash 嵌入)设施。理由:服务端验证是权威,客户端自校验 ROI 低 + 阻碍合法审计"
- `Cargo.toml` 无 build.rs 配置,无 .text 段 hash 相关依赖
- git commit `4731300 refactor(sdk): Day 1 - 撤掉反逆向红线设施 + 配置 NDK 交叉编译` 是明确证据

但 ADR 0058 状态仍是 `accepted`,与代码事实严重不符,违反 CLAUDE.md 第 10 节。本 ADR 补齐决策追溯。

## 决策

### v2 撤除 Rust so 自校验,服务端验证是权威

**撤除的能力:**

- ❌ `build.rs` 编译时计算 .text 段 hash 并嵌入常量
- ❌ JNI 入口分散校验 .text 段
- ❌ 服务端下发"预期 so hash"接口(从未实现)

**保留的能力:**

- ✅ APK 签名 hash 比对(ADR 0025/0062,但实现简化,见 ADR 0070)
- ✅ 服务端验证(activate/validate/heartbeat)作为唯一权威
- ✅ heartbeat 每 20 分钟轮换 AES 密钥(ADR 0060,已实现)

### 撤除理由

1. **ROI 低**:开源 + SaaS 双模式下,Rust 源码完全公开(ADR 0042),攻击者 100% 知道算法。so 自校验只能防"patch 单个 so",但 patch 自校验本身也是 patch so,只需多改一处。安全增益 < 实现成本。
2. **阻碍合法审计**:开源项目的安全审计员需要检查 so 行为,自校验会让审计员每次审计都要先绕过校验,降低社区信任(ADR 0042 公开边界原则)。
3. **服务端验证是权威**:ADR 0019 安全哲学明确"提高成本 + 检测异常 + 快速响应",服务端验证 + 风控 + 封禁响应才是核心防线。客户端任何校验最终都要回到服务端验证一次。
4. **编译时 hash 嵌入的实际效果**:hash 常量本身可被 patch(攻击者改 hash 常量 + 改 .text 段同步),自校验形同虚设。ADR 0058 已承认"hash 常量可 patch",组合方案未实际落地。

### 与 ADR 0025/0062 的关系

ADR 0025(完整性校验)三项中:
- (a) APK 签名校验:**保留**(实现简化,见 ADR 0070)
- (c) so 自校验:**撤除**(本 ADR)
- (d) 服务端下发校验值:**部分保留**(APK 签名白名单下发,见 ADR 0070;so hash 下发撤除)

### 与 ADR 0058 备选方案的关系

ADR 0058 备选方案"分散校验"和"服务端下发"两项组合,实际 v2 只保留服务端验证(无需下发 so hash),撤掉了所有客户端 .text 段校验。

## 备选方案

| 方案 | 优点 | 缺点 | 不选原因 |
|---|---|---|---|
| A. 恢复 ADR 0058 完整实现 | 双重校验 | ROI 低、阻碍审计、hash 可 patch | 实际效果不及成本 |
| B. 仅保留编译时嵌入,撤掉服务端下发 | 简单 | hash 常量可 patch,形同虚设 | 不安全 |
| C. 完全撤除,服务端验证权威(本方案) | 简单、可审计、与开源哲学一致 | 客户端无自校验 | 合理(服务端是权威) |
| D. 改用硬件信任根(TEE/Keymaster) | 最强 | 实现复杂、设备兼容性差 | MVP 过度设计 |

## 影响

- **正面:**
  - ADR 与代码事实一致(CLAUDE.md 第 10 节合规)
  - 编译流程简化(无 build.rs 依赖)
  - 开源审计友好(审计员无需绕过自校验)
  - 与 ADR 0019 安全哲学一致(服务端是权威)
- **负面:**
  - 客户端 so 被 patch 后无自检能力,完全依赖服务端风控
  - 离线缓存场景(ADR 0017)在缓存有效期内无法检测 so 被改
- **风险:**
  - 攻击者 patch so 后,在缓存有效期内(默认 7 天)可绕过验证
  - 缓解:缓存过期后必须联网验证,服务端可下发"强制下线"指令(ADR 0017,待实现)
  - 缓解:heartbeat 每 20 分钟轮换密钥(ADR 0060,已实现),长期离线才会触发缓存
