# ADR 0076 · 项目定位修订评估:攻防工作台

- 状态:**accepted**(方案 B,2026-07-19)
- 日期:2026-07-19
- 决策者:小城笺项目
- 层次:战略

## 决策(2026-07-19 accept)

**accept 方案 B:自有 APK 诊断(有限红方)**

用户决策理由:
- 认同 ADR 0076 对刑法 285 条风险的评估
- 小城笺作为独立项目,现阶段规避刑事风险是第一优先级
- 红方功能严格限定为"仅处理用户本地拥有的、具有合法著作权的 APK 文件"

**对外话术修正**:小城笺定位更新为——「独立开发者的私有应用攻防与遗产维护工具」。强调"私有"和"自有",彻底切割与盗版/黑产的关联。

**技术兜底要求**(法律自证清白的证据):
- 包名白名单:仅允许诊断在 admin-web 注册过的自有应用包名
- 签名校验比对:上传的 APK 签名必须与开发者后台配置的预期签名 hash 匹配
- 本地私有目录隔离:诊断操作在服务器临时目录 `/tmp/audit/<taskId>/` 完成,不持久化,处理后立即删除
- 三重校验任一失败即拒绝诊断

**法律咨询建议**:采纳,但在方案 B 框架下,主要是求"合规性确认",而非风险评估。

**取代(superseded by 0076)**:
- ADR 0001 · 项目定位与合规红线(状态改 superseded by 0076)

**功能级 ADR**:
- ADR 0077 · 自有 APK 诊断功能(含技术兜底实现细节)

## 背景

### 1. 用户请求

2026-07-19,项目 owner 提出"定位修正"请求,核心要点:

1. 将小城笺定位从"单纯的防盗版工具"修正为「面向独立开发者的应用攻防与商业化工作台」
2. 红方能力(脱壳、去签名校验)作为"攻防工作台"标配,纳入开发路线图
3. 红方正当用途:审计自有 SDK 后门 / 修复加固兼容性崩溃 / 竞品分析 / 遗产维护 / 对标 MT/NP 管理器行业惯例
4. 对外话术从"反盗版"改为"攻防一体,让创作者的代码既硬气又好用"
5. 功能优先级:基础层(红,脱壳+去签)< 核心层(红+蓝,注入+卡密)< 增值层(蓝,加固+防二次打包)

### 2. 与现有合规红线的冲突

本请求与以下项目文档直接冲突:

| 现有文档 | 现有条款 | 冲突点 |
|---|---|---|
| `CLAUDE.md` 第 2 节 | "不得实现绕过其他验证系统的功能" | 红方"去签校验"直接违反 |
| `CLAUDE.md` 第 2 节 | "不得在客户端硬编码任何通用绕过反作弊的逻辑" | 红方"通用脱壳"直接违反 |
| `CLAUDE.md` 第 2 节 | "如果用户要求实现上述功能,立即拒绝并提醒红线" | 本 ADR 即是"提醒红线 + 走 ADR 流程"的执行 |
| `ADR 0001` 项目定位 | "开源 + SaaS 双模式 Android 卡密验证系统,保障创作者权益,杜绝付费应用盗版" | 与"攻防工作台"定位不符 |
| `ADR 0001` 合规红线 | "禁止:用于外挂、私服、破解工具、规避反作弊" + "主动提供绕过其他验证系统的功能" | 红方功能直接违反 |

| `ADR 0068` | v2 已移除 dex 注入路径,改为 SDK 集成辅助 | v2 重构刚移除的能力,本请求要重新加回 |
| `docs/compliance/user-agreement.md` 第 2 节 | "禁止:主动提供绕过其他验证系统的功能 / 对他人 APK 进行重打包、注入、修改字节码" | 红方功能直接违反 |

### 3. 走 ADR 流程的必要性

CLAUDE.md 第 10 节强制规定:**所有重大决策必须写 ADR,决策变更写新 ADR 标记旧 ADR 为 superseded**。

项目定位变更 + 合规红线修订是最重大的战略决策,必须:
- 起草新 ADR(本 ADR 0076)评估法律风险 + 列多方案
- 用户 review + 法律咨询后决定 accept/reject
- 若 accept,同步修订 CLAUDE.md / ADR 0001 / README / user-agreement.md / 相关 ADR
- 若 accept 方案 A(通用红方),后续还需功能级 ADR(0077 脱壳 / 0078 去签)

**注**:上任工程师多次违反第 10 节(代码推翻 ADR 不写新 ADR),本 ADR 严格遵守,先评估再决策。

## 决策候选(3 方案,待用户选择)

### 方案 A:通用红方(对标 NP 管理器)

**功能范围**:
- 通用脱壳:集成 FRIDA-DEXDump 或类似工具,任意 APK 可脱壳
- 通用去签名校验:对任意 APK 去除签名校验逻辑(Java/Native 层)
- 基础层定位:标配功能,非 VIP 独占

**实现路径**:
- 后端:`backend/src/red-team/` 新模块(脱壳服务 + 去签服务)
- 客户端:`injector-android/` 加"红方工具"Tab
- CLI:`injector unpack` / `injector strip-sign` 子命令
- 强制免责协议:首次使用前签电子协议
- 审计日志:所有红方操作记录到 PG `red_team_audit_log` 表

**法律风险**:**高**
- 《著作权法》第 48 条:避开技术措施(民事 + 行政责任)
- 《计算机软件保护条例》第 24 条:故意删除/改变权利管理信息(行政 + 民事)
- 刑法 285 条:提供侵入/非法控制计算机信息系统程序、工具罪(**刑事风险**,视情节)
- 民法侵权:被加固厂商 / APK 著作权人起诉赔偿

**免责协议**(强制):
1. 用户首次使用红方功能前强制签电子协议
2. 协议明确:用户保证拥有 APK 著作权或合法授权
3. 协议明确:用户承担一切法律后果
4. 协议明确:平台保留配合司法调查权利
5. 签署记录审计日志(保留 1 年)
6. 每次红方操作再次确认(防误操作)

**对外话术**:攻防一体,让创作者的代码既硬气又好用

**用户协议修订**:
- 第 2 节"禁止的使用"中删除"主动提供绕过其他验证系统的功能"
- 新增第 3 节"红方功能特别声明"(参照 ADR 0029 模板)
- 第 6 节"付费与退款"加红方功能付费条款

**优势**:
- 满足"对标 NP 管理器"完整诉求
- 覆盖用户列出的 5 项红方正当用途(部分成立)
- 行业惯例对齐

**劣势**:
- 法律风险显著,可能被加固厂商起诉
- 刑法 285 条风险(若被认定为"提供侵入工具")
- 项目品牌可能被定性为"灰产工具链"
- SaaS 服务器持有他人 APK = 侵权证据(ADR 0068 已排除,v2 已删除 inject 模块,本方案要重新加回)

### 方案 B:自有 APK 诊断(有限红方)

**功能范围**:
- 仅对开发者**自有** APK 做反编译查看(JADX 类静态分析)
- 需上传签名证书证明所有权(校验 APK 签名与上传证书匹配)
- 不提供通用脱壳 / 通用去签
- 不修改字节码,只读查看

**实现路径**:
- 后端:`backend/src/audit/` 新模块(自有 APK 诊断)
- 客户端:`injector-android/` 加"自有诊断"Tab
- CLI:`injector audit --apk <path> --cert <signing-cert>`(需证书证明所有权)
- 类似 Android Studio APK Analyzer + JADX 静态查看

**法律风险**:**低**
- 自有 APK 反编译查看是合法的(开发者对自己 APK 有完整权利)
- 不涉及避开技术措施(只读,不修改)
- 不涉及提供通用绕过工具

**对外话术**:自有资产审计,看清自己的代码

**用户协议修订**:
- 第 2 节保持不变(仍禁止通用绕过)
- 新增第 3 节"自有 APK 诊断"条款
- 不需免责协议(自有 APK 无需免责)

**优势**:
- 满足用户列出的 2 项成立正当用途(审计自有 SDK + 遗产维护)
- 法律风险低
- 不需免责协议,接入门槛低
- 与 v2 重构(ADR 0068)边界一致

**劣势**:
- 不满足"对标 NP 管理器"的完整红方诉求
- 不覆盖"竞品分析"(本身不成立)
- 不覆盖"修复加固兼容性崩溃"(边际场景)

### 方案 C:维持现状(拒绝变更)

**功能范围**:
- 不引入红方功能
- 继续 ADR 0001 定位(开源 + SaaS 双模式 Android 卡密验证系统)
- 继续 CLAUDE.md 第 2 节红线
- v2 重构(ADR 0068)边界不变

**法律风险**:**零**

**对外话术**:不变(保障创作者权益,杜绝付费应用盗版)

**优势**:
- 零法律风险
- 零开发成本
- 与 v2 重构 + 现有 ADR 一致
- 已完成的 P0-P5 工作不需返工

**劣势**:
- 不满足用户本次定位变更诉求
- 不覆盖红方正当用途(自有 SDK 审计需用其他工具)

## 红方正当用途逐项评估

用户列出 5 项红方正当用途,逐项评估:

| # | 用途 | 评估 | 法律风险 | 是否需红方功能 |
|---|---|---|---|---|
| 1 | 审计自己集成的第三方 SDK 是否有后门 | ✅ 成立 | 低(自有 APK) | 方案 B 即可(JADX 反编译查看) |
| 2 | 修复特定加固环境下的兼容性崩溃 | ⚠️ 边际 | 中(需脱壳) | 方案 A 可覆盖,但场景小众 |
| 3 | 竞品分析(学习行业优秀实现) | ❌ 不成立 | 高(逆向他人 APK 违法) | 任何方案都不应支持 |
| 4 | 遗产维护(自己已丢失源码的旧应用) | ✅ 成立 | 低(自有 APK) | 方案 B 即可 |
| 5 | 行业惯例(MT/NP 管理器红蓝一体) | ⚠️ 不等于合法 | - | 行业惯例不构成合法理由 |

**结论**:5 项理由中,2 项成立(自有 SDK 审计 + 遗产维护),2 项边际,1 项不成立。

- 方案 B(自有 APK 诊断)可完整覆盖 2 项成立用途
- 方案 A(通用红方)额外覆盖 2 项边际用途,但引入高法律风险
- 竞品分析在任何方案下都不应支持

## 法律风险逐项评估(方案 A 适用)

### 1. 《中华人民共和国著作权法》第 48 条

**条款**:未经著作权人许可,故意删除或者改变作品、录音录像制品等的权利管理信息的,法律、行政法规另有规定的除外。

**适用**:去签名校验 = 删除/改变权利管理信息

**责任**:
- 民事:赔偿损失
- 行政:著作权行政管理部门责令停止侵权,没收违法所得,罚款
- 刑事(情节严重):刑法第 217 条侵犯著作权罪

### 2. 《计算机软件保护条例》第 24 条

**条款**:未经软件著作权人许可,故意删除或者改变软件的权利管理信息的,构成侵权。

**适用**:去签 + 脱壳均可能触发

**责任**:同上

### 3. 《中华人民共和国刑法》第 285 条

**条款**:提供专门用于侵入、非法控制计算机信息系统的程序、工具,或者明知他人实施侵入、非法控制计算机信息系统的违法犯罪行为而为其提供程序、工具,情节严重的,处三年以下有期徒刑或者拘役,并处或者单处罚金;情节特别严重的,处三年以上七年以下有期徒刑,并处罚金。

**适用**:通用脱壳 + 去签工具若被认定为"专门用于侵入/非法控制计算机信息系统的程序、工具"

**责任**:**刑事**(三年以下 / 三年以上七年以下有期徒刑 + 罚金)

**风险点**:
- "专门用于"的认定:若工具无合法用途(或合法用途占比极低),可能被认定为"专门用于"
- "情节严重":提供次数 / 违法所得 / 传播范围
- 司法实践:类似工具(如"看雪"系列)曾有判例

### 4. 民法侵权

**适用**:被加固厂商(360/腾讯/爱加密/梆梆)或 APK 著作权人起诉

**责任**:
- 停止侵权
- 赔偿损失(含合理维权费用)
- 消除影响

### 5. 风险综合评估

| 方案 | 著作权法 48 条 | 软件保护条例 24 条 | 刑法 285 条 | 民法侵权 | 综合风险 |
|---|---|---|---|---|---|
| A(通用红方) | 高 | 高 | 中-高 | 高 | 🔴 高 |
| B(自有诊断) | 低 | 低 | 低 | 低 | 🟢 低 |
| C(维持现状) | 无 | 无 | 无 | 无 | ✅ 零 |

## 免责协议草案要点(方案 A 适用)

若用户选方案 A,免责协议需含:

1. **用户保证**:
   - 对所脱壳/去签的 APK 拥有著作权,或
   - 已获得 APK 著作权人的书面授权
   - 承担因使用红方功能产生的一切法律后果

2. **平台权利**:
   - 保留配合司法调查的权利
   - 保留在收到合法投诉时暂停或终止用户账号的权利
   - 保留修改或下线红方功能的权利

3. **签署流程**:
   - 首次使用红方功能前强制签电子协议
   - 协议明确 4 项条款(著作权保证 / 法律后果承担 / 配合司法 / 平台权利)
   - 签署记录审计日志(保留 1 年)
   - 每次红方操作再次确认(防误操作)

4. **审计日志字段**:
   - developerId / 操作类型 / APK hash / 时间戳 / IP / userAgent
   - 卡密明文绝不进日志(ADR 0027 强制)

5. **APK 处理**:
   - 上传的 APK 不持久化(处理完立即删除,ADR 0028 原则)
   - 服务器临时存储 `/tmp/red-team/<taskId>/`,完成后删除
   - 注:这与 ADR 0068 v2 移除"服务器端 APK 处理"冲突,需重新评估

## 影响

### 正面影响

- **方案 A**:
  - 满足用户完整红方诉求
  - 行业惯例对齐(MT/NP 管理器)
  - 可能吸引灰产用户(短期收入增长)
- **方案 B**:
  - 满足 2 项成立正当用途(自有 SDK 审计 + 遗产维护)
  - 法律风险可控
  - 与 v2 重构边界一致
- **方案 C**:
  - 零风险
  - 已完成 P0-P5 工作不返工

### 负面影响

- **方案 A**:
  - 法律风险显著(刑法 285 条 + 民事 + 行政)
  - 可能被加固厂商起诉
  - 项目品牌可能被定性为"灰产工具链"
  - v2 重构(ADR 0068)部分返工
  - 服务器持有他人 APK = 侵权证据
- **方案 B**:
  - 不满足"对标 NP 管理器"完整诉求
  - 用户可能失望
- **方案 C**:
  - 用户本次定位变更诉求未满足

### 风险

- **方案 A**:
  - 刑事风险:若被认定为"提供侵入工具",项目 owner 可能承担刑事责任
  - 民事风险:被加固厂商 / APK 著作权人起诉
  - 品牌风险:被开源社区抵制(开源项目提供灰产工具)
  - 运营风险:服务器被查封 / 域名被冻结
- **方案 B**:
  - 边际风险:用户上传他人 APK 试图诊断(需证书校验防)
- **方案 C**:
  - 无新增风险

## 决策建议

**我作为首席架构师与安全顾问的建议**:

**推荐方案 B(自有 APK 诊断)**,理由:

1. 覆盖用户列出的 2 项成立正当用途(自有 SDK 审计 + 遗产维护)
2. 法律风险低(自有 APK 反编译合法)
3. 与 v2 重构(ADR 0068)边界一致,不返工
4. 不需免责协议,接入门槛低
5. 不影响项目品牌(开源社区接受)

**不推荐方案 A**(通用红方),理由:

1. 刑法 285 条风险不可控(项目 owner 个人可能承担刑事责任)
2. 项目品牌被定性为"灰产工具链"的风险
3. v2 重构返工成本高
4. 5 项正当用途中 3 项不成立或边际,不足以支撑高法律风险
5. 行业惯例不构成合法理由

**方案 C**(维持现状)是次推荐,若用户对方案 B 仍不满意。
