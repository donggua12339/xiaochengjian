# ADR 0079 · 部分取代 ADR 0067(仅"自动脱壳推 v3"条款,梆梆自检场景)

- 状态:accepted(律师意见已落地,2026-07-20,与 ADR 0078 同步生效)
- 日期:2026-07-20
- 决策者:小城笺项目
- 层次:功能 / 合规
- Supersedes: ADR 0067(partial,仅"自动脱壳推 v3"条款,仅梆梆自检场景)

## 背景

ADR 0067(2026-07-14,accepted)定调:
- MVP 不支持加固 APK,壳 dex 注入放 v2
- **自动脱壳延后到 v3**(需免责协议)

ADR 0077(2026-07-19,accepted)引入"自有 APK 诊断",其例外 B(2026-07-20 修订)允许对自有梆梆加固 APK 做加固层自检(完整性报告,不脱壳不反编译),具体实现见 ADR 0078(2026-07-20,proposed,律师前置)。

ADR 0078 的梆梆自检功能与 ADR 0067 的"自动脱壳推 v3"条款在梆梆这一家厂商的自有 APK 场景下产生部分冲突:梆梆自检允许"对加固层做完整性扫描"(不脱壳),而 ADR 0067 原文把所有加固相关操作一律推到 v3。

需明确:**ADR 0067 的"自动脱壳推 v3"条款,在梆梆自检场景下被本 ADR 部分取代**;其他条款(MVP 拒绝加固 APK、壳 dex 注入推 v2、运行时 hook 拒绝)以及其他加固厂商(360/爱加密/乐固等),仍维持 ADR 0067 原决策,推迟至 v3 评估。

## 决策

**ADR 0079 部分取代 ADR 0067,范围仅限 ADR 0067 的"自动脱壳推 v3"条款,且仅对"自有梆梆加固 APK 的加固层自检"场景生效。**

### 取代范围(仅一条)

ADR 0067 原文:
> v2:壳 dex 注入(LSPatch 类方案)+ 自动脱壳(需免责协议)

本 ADR 在梆梆自检场景下,将上述条款中的"自动脱壳"部分改为(律师审核原文,一字不差):

> v2 周期内,允许蓝方-自有梆梆加固层自检适配器(ADR 0078),限定:仅梆梆 / 仅 integrity 报告 / 不解未知加固。其余加固类型(360/爱加密/乐固)仍维持 0067 原决策,推迟至 v3 评估。

### 不取代范围(ADR 0067 以下条款仍完全有效)

1. **MVP 拒绝加固 APK**(ADR 0067 §决策第 1 条):非梆梆加固 APK 仍拒绝
2. **壳 dex 注入推 v2**(ADR 0067 §决策第 2 条前半):本 ADR 不涉及
3. **运行时 hook(LSPatch 类)拒绝**:本 ADR 不涉及
4. **其他加固厂商的脱壳/自检**:仍按 ADR 0067 推 v3 评估
5. **梆梆加固 APK 的非自检操作**(注入 / 修改字节码 / 重打包 / 通用脱壳):仍按 ADR 0067 拒绝或推 v3

### 取代的前提(任一不满足则 ADR 0067 该条款仍完全有效)

1. **律师意见落地**:ADR 0078 §5 律师意见书通过,ADR 0078 状态从 proposed 改为 accepted
2. **ADR 0078 3 把锁全部到位**:锁 A(仅梆梆)+ 锁 B(EULA 前置)+ 锁 C(仅完整性报告,不输出源码)
3. **三重校验通过**:ADR 0077 §2 三重校验(包名白名单 + 签名 hash 比对 + 目录隔离)

**前提不满足时的回退**:本 ADR 自动失效,ADR 0067 "自动脱壳推 v3"条款恢复完全有效,梆梆适配器代码删除或保留但不启用(由用户决定)。

## 备选方案

| 方案 | 含义 | 不选原因 |
|---|---|---|
| A. 完全取代 0067 | 所有加固 APK 都允许自检 | 违反 ADR 0078 锁 A(仅梆梆),法律风险高 |
| B. 部分取代 0067 仅"自动脱壳推 v3"条款(本方案) | 梆梆自检场景取代,其他条款不变 | 用户决策 + 律师前置 + 最小取代范围 |
| C. 不取代,新建独立 ADR 0080 | 不动 0067,加 0080 允许梆梆 | 决策关系混乱,违反 ADR 追溯规范(CLAUDE.md §10) |
| D. 修改 0067 原文 | 直接改 0067 内容 | 违反 CLAUDE.md §10(ADR 不可变,只能 superseded) |
| E. 取代 0067 多条条款 | 连"MVP 拒绝加固"也取代 | 范围过大,其他加固厂商仍需拒绝 |

## 影响

### 正面影响

- 最小取代范围:只动 0067 的"自动脱壳推 v3"一条,其他条款不变,追溯清晰
- 限定梆梆场景的法律风险可控(3 个限定:仅梆梆 / 仅 integrity / 不解未知)
- 前提条件回退机制提供安全网
- 与 ADR 0077 例外 B + ADR 0078 形成完整合规链

### 负面影响

- 取代前提复杂(律师意见 + 3 把锁 + 三重校验),需在代码中明确实现
- 梆梆适配器代码需在 hardener-detector 严格区分梆梆 vs 其他加固,误判会违反取代范围

### 风险

- **取代范围误判**:开发时把非梆梆加固 APK 也走了 0078 路径
  - 缓解:hardener-detector 严格匹配梆梆特征,非梆梆直接返回 `UNSUPPORTED_HARDENER`,与 ADR 0067 拒绝路径一致
- **律师意见被驳回**:ADR 0078 rejected,本 ADR 同步失效
  - 缓解:回退机制已定义,代码可一键禁用梆梆适配器
- **取代条款误读**:把本 ADR 理解为取代整个 0067
  - 缓解:本 ADR 显式标注 Supersedes 范围为"仅自动脱壳推 v3 条款",并列出 5 项不取代范围
