# ADR 0082-A · 360 加固保自检适配器(V1.5b)

- 状态:proposed(合规核查已通过,可开发)
- 日期:2026-07-21
- 决策者:小城笺项目
- 层次:功能 / 安全 / 合规
- 关联 ADR:0078(梆梆适配器,锁 A 扩展)

## 背景

ADR 0078 锁 A 原文:"仅梆梆一家,360 / 爱加密 / 乐固 / 腾讯乐固等其他加固方案明确不支持"。

V1.5 阶段扩展加固自检范围,基于市场调研(CSDN Android 加固平台横评),国内独立开发者加固使用率 Top 3 为:梆梆、360 加固保、腾讯乐固,合计覆盖约 90% 目标用户。

本 ADR 扩展 ADR 0078 锁 A 的支持范围,增加 360 加固保。

## 合规核查结论(已确认,2026-07-21)

### 核查依据

- 《360 加固助手软件产品许可使用协议》第 4.1 条
- 《360 用户服务协议》第 4.2 条 + 第 3.15 条

### 逐条分析

**第 4.1 条原文**:"不得对本软件及其包含的任何部分进行反向工程"。

**关键定性**:这里的"本软件"特指 **360 加固助手程序本身**(即 jiagu.jar / 360Jiagu 工具),**绝不指代**开发者上传的、经过加固的自有 APK 文件。

**第 3.15 条(开发者使用规则)**:仅限制了"上传木马、非法用途、侵犯版权"等行为,全文未提及禁止开发者对已加固的 APK 进行"完整性自检"或"安全审计"。

**结论**:360 授予了开发者使用加固服务的权利,但未剥夺开发者对自己 APK 进行后续安全运维的权利。

### 与 ADR 0078 的同构性

ADR 0078 对梆梆加固的处理逻辑:"对开发者自有 APK 的加固层进行自检,不属于对梆梆软件的反向工程"。

360 场景完全适用同一法律逻辑:小城笺处理的是 `com.xxx.app`(开发者资产),而非 `com.qihoo.jiagu`(360 资产)。

### 最终裁定

360 协议第 4.1 条**不构成**对 ADR 0082-A 的阻碍。"反向工程"的禁令不适用于"对自己拥有的、已加固的 APK 进行完整性校验"。

**合规风险已排除,无需律师前置,可直接进入开发。**

## 决策

### 360 加固保特征

| 特征 | 值 |
|---|---|
| so 文件 | libjiagu.so(360 加固标志) |
| Application 类名前缀 | com.qihoo.util.* / com.qihoo.* |
| 加固产物 | assets/libjiagu.so + 原 dex 加密 |

### 锁 A 扩展(本 ADR)

ADR 0078 锁 A 从"仅梆梆"扩展为"梆梆 + 360 加固保"。其他锁(B/C + ADR 0081 锁 7)不变。

### 检测规则

```typescript
// 360 加固保 so 特征(从 UNSUPPORTED 列表移到支持列表)
const QIHOO_360_SO_PATTERNS = [
  /^lib\/[^/]+\/libjiagu\.so$/i,
];

const QIHOO_360_APP_PREFIXES = ['com.qihoo.util.', 'com.qihoo.'];
```

### EULA 前置(锁 B)

360 加固保独立 EULA 文本(`docs/compliance/audit-eula-360.md`,待起草),开发者备案时强制勾选。

EULA 核心条款:"本人为 APK 著作权人,本次自检系对自有资产的安全审计"。

### 完整性报告(锁 C)

输出字段与梆梆适配器一致(soFiles / entryClass / signatures / suspiciousCalls / scanVersion / scanTime),不含源码。

### 技术实现

- 复用 HardenerDetector 逻辑,检测 libjiagu.so 和 com.qihoo.util 等 360 特征
- 适配器仅输出完整性报告(so hash、签名状态),严禁触碰 dex 或输出源码(维持锁 C)
- EULA 文本只需开发者确认:"本人为 APK 著作权人,本次自检系对自有资产的安全审计"
