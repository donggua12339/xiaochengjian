# ADR 0085 · 加固 APK 与 xcj-auth-sdk 的兼容性边界分析

- 状态:accepted(边界分析,无代码实现)
- 日期:2026-07-21
- 决策者:小城笺项目
- 层次:架构 / 合规

## 1. 技术死锁的形式化证明

### 前提

**P1**:Android 7.0+ 强制 V2 签名,9.0+ 强制 V3 签名。V2/V3 对整个 APK 字节做完整性校验,修改 Manifest / assets / dex / resource 任一文件必然使签名失效,必须重签。

**P2**:加固壳(360/梆梆/乐固)在运行时 Native 层校验 APK 签名。加固前后 keystore 不一致 → 壳主动 Crash,识别为二次打包。

**P3**:即使不重签(保留原签名),修改了 APK 内容 → V2/V3 校验失败 → 系统拒绝安装。

**P4**:注入 SDK 必须修改 APK 内容(加 dex / 改 Manifest)。

### 推导

```
P4: 注入 SDK 必须改 APK 内容
  ↓
P1: 改内容 → 签名失效 → 必须重签
  ↓
P2: 重签 → keystore 与加固时不一致 → 壳闪退
  ↓
P3: 不重签 → 系统拒绝安装
  ↓
结论: 三者死锁,无解。
```

```
加固 APK + 注入 SDK
  ├── 改内容 + 重签 → 壳闪退(P2)
  ├── 改内容 + 不重签 → 系统拒绝安装(P3)
  └── 不改内容 → 无法注入 SDK(P4)
```

**形式化结论**:在"不解固、不重签、不改 APK 结构"的前提下,加固 APK 注入 xcj-auth-sdk **物理上不可能**。

## 2. 合规替代路径

为"已加固 APK + 丢 keystore"的开发者,提供以下合规出路:

### 路径 A:官方加固厂商协作(B2B)

小城笺提供 xcj-auth-sdk 的 aar 包,主动对接梆梆/360/腾讯乐固,提议"官方加固流程中集成 xcj-auth-sdk"合作模式。加固厂商在自己的加固 pipeline 里集成 SDK,输出"已加固 + 已集成 SDK"的 APK,小城笺不碰加固 APK。

- **合规**:✅ 完全合规。加固厂商是加固层的权利方,合作即授权。
- **实施**:商务拓展,非技术开发。
- **小城笺职责**:提供 SDK aar + 集成文档,不参与加固流程。

### 路径 B:未加固 APK 封装 + 开发者自加固(推荐)

1. 开发者提供**未加固 APK** → 走 ADR 0081 封装(注入 SDK + 自备 keystore 重签)
2. 开发者用加固厂商官网/工具对封装后的 APK **自行加固**
3. 加固时使用与"封装重签"**相同的 keystore**(开发者自备)

- **合规**:✅ 完全合规。开发者自加固,小城笺不碰加固流程。
- **技术**:✅ 可行。360 官方文档明确"加固前后用相同 keystore 重签"即可避免闪退。
- **流程**:
  ```
  未加固 APK → Packer 封装(注入 SDK + keystore-A 重签)
  → 加固官网加固(用 keystore-A 重签) → 分发
  ```

### 路径 C:源码级集成(ADR 0080)

开发者在 Android Studio 中集成 xcj-auth-sdk 源码依赖,编译出 APK 后再自行加固。

- **合规**:✅ 完全合规。标准 SDK 集成,零字节码修改。
- **技术**:✅ 可行。适用于有源码的开发者。

### 路径 D:换包名重新发布(灰度)

开发者用新 keystore 对"解固清理后的 APK"(若通过合法途径获得未加固版)重签,换包名重新上架。

- **合规**:⚠️ 灰度。解固需合法途径(如加固厂商协助),不建议小城笺参与解固。
- **技术**:✅ 可行。开发者自行操作。

## 3. 小城笺侧可提供的辅助工具(不踩红线)

虽然不能直接做"加固 APK 注入 SDK",但可以提供:

| 工具 | 说明 | 状态 |
|---|---|---|
| keystore 生成向导 | 帮助丢 keystore 的开发者生成新 keystore | ADR 0081 扩展 |
| 未加固 APK 封装服务 | 路径 B 的第 1 步(injector sign / Packer) | ✅ 已有(ADR 0081) |
| 加固厂商对接文档 | 为开发者提供"360/梆梆/乐固 官方加固 + 集成 xcj-auth-sdk"的操作手册 | 待写 |
| xcj-bundle 分发格式 | 将"未加固封装 APK + 加固配置"打包,开发者下载后一键调用加固厂商 CLI 完成加固 | 可选(未来) |

## 4. 明确不作为

小城笺**不会**实现以下能力:

- ❌ 通用脱壳(任何加固类型,包括但不限于梆梆/360/乐固)
- ❌ 加固 APK 的 SDK 注入封装(物理不可行 + 合规红线)
- ❌ 加固壳的签名校验绕过(违反《著作权法》第 48 条)
- ❌ 加固层的运行时 Hook/修补(违反 ADR 0076 方案 B)
- ❌ 加固 APK 的"解固→封装"链路(违反 ADR 0081 但书 C)

## 6. 影响

### 正面影响

- 以文档形式锁定"加固 APK + SDK 注入"的技术边界,避免后续重复讨论
- 为开发者提供 4 条合规替代路径,在合规框架内最大化服务能力
- 与所有现有 ADR 兼容,无代码改动

### 负面影响

- "丢 keystore + 已加固 APK"场景无解(物理定律,非小城笺能解决)
- 路径 A(B2B 合作)需商务推动,非技术可控
