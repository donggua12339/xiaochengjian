# ADR 0087 · 深度安全审计与漏洞赏金赋能计划

- 状态:proposed(设计阶段,代码未实现)
- 日期:2026-07-21
- 决策者:小城笺项目
- 层次:功能 / 安全 / 合规

## 背景

小城笺已具备:

- ADR 0077:自有 APK 只读诊断(JADX 反编译查看 + 签名信息 + SDK 后门扫描)
- ADR 0078/0082:梆梆/360/腾讯乐固加固层完整性自检(仅报告,不出源码)
- ADR 0085:加固 APK 与 SDK 兼容性边界分析(明确"加固 APK 不进 Packer")

现有的"自检"报告停留在基础层面(so 文件 SHA-256 + 签名状态 + 权限列表)。随着《个人信息保护法》实施及供应链攻击频发,市场对 APK 深度安全审计的需求激增。

本 ADR 将现有的"逆向解析能力"升级为"深度安全审计引擎",并拓展漏洞赏金辅助研究功能。**全程只读,不改字节码,不重打包。**

## 决策

### 红线重申(不可逾越)

1. **只读原则**:所有分析在内存中进行,严禁将解固/反编译的中间产物落盘超过 5 分钟(审计日志除外),严禁提供下载接口
2. **无修改原则**:严禁任何形式的字节码修改、资源替换或重打包逻辑;输出仅为 JSON 格式的审计报告或只读代码高亮视图
3. **依赖现有架构**:复用 HardenerDetector 的文件解析能力,复用 audit_log_own 的审计存储机制

### 模块一:隐私合规与供应链安全审计(方向 A)

利用逆向分析能力,在不修改 APK 的前提下,对 APK 进行深度体检。

#### 1.1 隐私合规雷达

| 检测项 | 方法 | 说明 |
|---|---|---|
| 权限-API 匹配 | 扫描 AndroidManifest.xml 声明的权限 vs DEX 代码中的实际调用 | 标记"声明但未使用"或"使用但未声明"的权限 |
| 高风险 API 检测 | 扫描 DEX 中调用 getMacAddress() / getDeviceId() / getSubscriberId() 等 API | 检测未被隐私政策披露的敏感信息采集 |
| 超范围收集 | 参照 GB/T 35273 标准,比对权限声明与实际业务功能 | 标记"超出业务必要范围"的权限 |
| 第三方 SDK 隐私 | 检测常见 SDK(友盟/极光/微信等)的隐私声明一致性 | 标记 SDK 采集行为与 APP 隐私政策不一致 |

**输出**:隐私合规评分(0-100)+ 风险项列表(严重/高/中/低)

#### 1.2 供应链投毒检测

| 检测项 | 方法 | 说明 |
|---|---|---|
| SDK 版本 CVE 检测 | 建立常见第三方 SDK Hash 特征库,比对已知 CVE 数据库 | 标记"含高危漏洞"的 SDK 版本 |
| Native 脆弱函数扫描 | 扫描 SO 文件中 strcpy / system / sprintf / gets 等脆弱函数调用 | 提示栈溢出/命令注入风险 |
| 恶意 Payload 检测 | 扫描 DEX/SO 中已知恶意特征(如远控/挖矿/勒索) | 标记可疑代码段 |
| 证书固定检测 | 检测是否使用证书固定(Certificate Pinning),防止 MITM | 给出加固建议 |

**输出**:供应链风险评分(0-100)+ CVE 列表 + 风险项列表

#### 1.3 加固有效性评估

结合 ADR 0082 加固检测,不仅检测"是否有壳",更进一步评估壳的配置强度:

| 检测项 | 方法 | 说明 |
|---|---|---|
| anti-debug 状态 | 扫描壳 so 中 ptrace / TracerPid 相关调用 | 评估反调试是否开启 |
| 签名校验强度 | 检测壳 so 中签名校验逻辑(判断 hash 比对方式) | 弱校验(仅比对 md5)/强校验(SHA-256) |
| 反 Hook 保护 | 检测 Xposed/Substrate/Frida 检测逻辑 | 评估反 Hook 能力 |
| 资源加密 | 检测 assets/ 下是否加密,加密算法强度 | 评估资源保护 |

**输出**:加固评分(0-100)+ 优化建议列表

### 模块二:漏洞赏金辅助研究平台(方向 B)

将小城笺打造为安全研究员(白帽)的辅助工具,帮助他们快速筛选目标 APK 中的潜在漏洞点。

#### 2.1 攻击面枚举

| 检测项 | 方法 | 说明 |
|---|---|---|
| Exported 组件 | 提取 AndroidManifest 中所有 `android:exported="true"` 的 Activity/Service/Receiver/Provider | 列出所有对外暴露的组件 |
| 开放端口 | 扫描 DEX 中 ServerSocket / DatagramSocket 绑定 | 标记潜在的网络攻击面 |
| WebView 风险 | 检测 `setJavaScriptEnabled(true)` + `addJavascriptInterface` 组合 | 标记 JS 注入风险 |
| 文件暴露 | 检测 FileProvider / openFile 等文件共享配置 | 标记路径遍历风险 |

**输出**:攻击面报告(组件列表 + 风险等级)

#### 2.2 静态污点分析雏形

| 检测项 | 方法 | 说明 |
|---|---|---|
| Source 识别 | 追踪 getIntent() / getExtras() / getQueryParameter() 等数据源 | 标记外部输入点 |
| Sink 识别 | 追踪 Runtime.exec() / startActivity() / sendBroadcast() 等敏感操作 | 标记危险操作点 |
| 数据流追踪 | Source → 中间变量 → Sink 的路径分析 | 标记潜在注入路径 |
| 常见漏洞模式 | Intent 注入 / 路径遍历 / 命令注入 / SQL 注入 | 按模式匹配 |

**输出**:污点分析报告(Source-Sink 路径 + 风险等级)

#### 2.3 CTF 训练场模式

| 功能 | 说明 | 限制 |
|---|---|---|
| DEX 反编译伪代码 | 类似 JADX 简化视图,高亮 CTF 考点 | 只读,禁止导出 |
| Smali 混淆提示 | 高亮混淆后的类名/方法名,提示还原思路 | 只读,不自动还原 |
| Native 层注册函数 | 列出 JNI_OnLoad 中注册的 native 函数 | 只读 |
| 算法逆向入口 | 标记加密/解密/哈希函数调用,辅助算法分析 | 只读,不自动破解 |

**输出**:CTF 学习报告(高亮区域 + 提示)

### 架构设计

```
backend/src/audit-own/security-analyzer/
├── security-analyzer.module.ts     # 模块注册
├── security-analyzer.service.ts    # 主流程(编排)
├── privacy-scanner.ts              # 模块一:隐私合规雷达
├── supply-chain-scanner.ts         # 模块一:供应链投毒检测
├── hardener-validator.ts           # 模块一:加固有效性评估
├── attack-surface-enumerator.ts    # 模块二:攻击面枚举
├── taint-analyzer.ts               # 模块二:静态污点分析
├── ctf-trainer.ts                  # 模块二:CTF 训练场
└── security-report.ts              # 报告生成(JSON + HTML)
```

### API 端点

| 端点 | 方法 | 作用 | 鉴权 |
|---|---|---|---|
| `/v1/audit/security-scan` | POST | 上传 APK,执行深度安全扫描(模块一+二) | JWT |
| `/v1/audit/security-scan/:taskId` | GET | 查询扫描状态/结果 | JWT |
| `/v1/audit/security-scan/history` | GET | 查询历史扫描记录 | JWT |

### 审计日志扩展

`audit_log_own` 表加字段:

| 字段 | 类型 | 说明 |
|---|---|---|
| securityScanType | String? | 扫描类型:`privacy` / `supply-chain` / `attack-surface` / `full` |
| securityScore | Int? | 安全评分(0-100) |
| securityFindings | Json? | 安全发现列表(JSON) |

### EULA 扩展

`HardenerEulaService` 加"深度安全审计"条款:

> 深度安全审计 EULA 条款:
> 1. 本人为 APK 著作权人,本次审计系对自有资产的安全自查
> 2. 审计全程只读,不修改 APK 内容,不输出反编译源码
> 3. 审计发现的漏洞/风险仅供本人参考,小城笺不承担因漏洞泄露导致的任何责任
> 4. 审计报告中的漏洞信息不得用于非法攻击他人系统

### 商业模式

深度安全审计功能作为 **Premium(高级)付费功能**:

| 版本 | 功能 |
|---|---|
| 免费版 | 基础诊断(ADR 0077)+ 加固自检(ADR 0078/0082) |
| 付费版(¥128/终身 或 ¥18/月) | 深度安全审计(模块一+二)+ SDK 封装(ADR 0081) |

## 影响

### 正面影响

- 隐私合规检测是 App 上架应用市场的刚需(Premium 付费功能)
- 吸引安全研究员使用小城笺,形成安全社区生态
- 与 ADR 0078/0082 加固自检形成"安全体检"闭环

### 负面影响

- 开发量大(2-3 周),需分阶段实现
- CTF 训练场需维护漏洞库(可持续更新成本)
- 隐私合规规则库需持续更新(GB/T 35273 等标准可能变化)

### 风险

- **漏洞报告泄露风险**:审计报告含漏洞信息,若被第三方获取可能被滥用
  - 缓解:审计报告仅开发者本人可查看(JWT),审计日志 1 年滚动清理
- **误报率**:静态分析可能产生误报(False Positive)
  - 缓解:报告标注"置信度",误报由开发者自行判断
- **规则库过时**:隐私合规标准可能变化
  - 缓解:规则库版本号 + 定期更新
