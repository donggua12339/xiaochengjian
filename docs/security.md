# 小城笺 · 安全白皮书

> 本文档公开小城笺卡密验证系统的安全设计架构、威胁模型、已知限制。
> 详见 [ADR 0042](adr/0042-security-disclosure-boundary.md) · 安全设计公开边界:架构公开,实现保留。

**最后更新**:2026-07-19

## 1. 安全哲学

小城笺是 **开源 + SaaS 双模式** 项目,客户端代码完全公开。我们承认根本约束:

| 事实 | 后果 |
|---|---|
| 客户端代码完全公开 | 攻击者 100% 知道算法 |
| 客户端在攻击者手里 | 任何 patch 都可能 |
| Rust so 也会被逆 | 只是难度高,不是不可能 |
| 服务端会被 CC/枚举 | 必须有风控 |

**我们的目标不是"防住所有攻击",而是"提高破解成本 + 检测异常 + 快速响应":**

1. **提高破解成本**:Rust so + 可选反逆向设施,让攻击者投入大量时间
2. **检测异常**:服务端风控检测枚举/异常设备/异常 IP
3. **快速响应**:封禁卡密 / 强制下线 / 拉黑设备

详见 [ADR 0019](adr/0019-security-philosophy.md)。

## 2. 五层防护架构

```
┌─────────────────────────────────────────────────────┐
│  第 1 层:通信安全(ADR 0020/0021/0060)              │
│  HTTPS + RSA-OAEP 握手 + AES-256-GCM + HMAC + nonce  │
│  每 20 分钟轮换 AES 密钥                              │
└─────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────┐
│  第 2 层:服务端安全(ADR 0022/0027)                  │
│  10 项安全基线强制 + IP/设备限流 + 失败锁定 + 错误延迟│
└─────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────┐
│  第 3 层:数据安全(ADR 0014/0018/0026)              │
│  卡密 SHA-256 hash + PostgreSQL RLS 多租户隔离 +     │
│  离线缓存 AES-256-GCM + HMAC 防篡改                  │
└─────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────┐
│  第 4 层:客户端核心(ADR 0023/0071)                  │
│  Rust so 加密/签名/缓存/校验 + 可选 obfstr +         │
│  可选 opaque-jni(默认透明,开发者按需启用)          │
└─────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────┐
│  第 5 层:完整性校验(ADR 0025/0070)                  │
│  APK 签名 hash 白名单比对 + 服务端下发(待 P2 补全)  │
└─────────────────────────────────────────────────────┘
```

## 3. 各层防护详解

### 3.1 第 1 层:通信安全

**已实现:**

| 机制 | 实现 | ADR |
|---|---|---|
| HTTPS 强制 | Cloudflare + 通配符证书 + 宿主 nginx 反代 | 0045 |
| RSA 握手 | 客户端用服务端 RSA 公钥加密临时 AES-256 密钥 | 0020 |
| AES-256-GCM 加密 | iv\|ciphertext\|tag,Base64 编码 | 0020 |
| HMAC-SHA256 签名 | method + path + timestamp + nonce + sha256(body) | 0021 |
| nonce 防重放 | Redis SETNX,5 分钟 TTL | 0021 |
| timestamp 偏差校验 | > 60s 拒绝 | 0021 |
| 密钥轮换 | 每 20 分钟 heartbeat 返回新 AES 密钥 | 0060 |

**握手流程:**

```
客户端                          服务端
  │                               │
  │ ── RSA 加密(临时 AES 密钥) ─> │
  │                               │ ── RSA 私钥解密 ──> 得 AES 密钥
  │ <───────── sessionId ──────────│
  │                               │
  │ ── AES 加密 + HMAC 签名 ─────> │(后续所有请求)
  │                               │ ── 验签 + 解密 + nonce 校验
  │ <───── AES 加密 + HMAC 签名 ───│
  │                               │
  │ ── heartbeat(每 20 分钟)──> │ ── 返回新 AES 密钥
  │                               │
```

### 3.2 第 2 层:服务端安全

**10 项安全基线(ADR 0027,生产模式强制):**

| 项 | 实现 |
|---|---|
| HTTPS 强制 | 启动检查,不允许 HTTP |
| 数据库密码强度 | 启动校验,弱密码拒绝启动 |
| 管理后台 2FA | TOTP,首次登录强制开启 |
| JWT 过期 | access 15min + refresh 7day |
| SQL 注入防护 | Prisma 参数化查询 |
| XSS 防护 | Vue 自动转义 + CSP 头 |
| CSRF 防护 | SameSite Cookie + CSRF token |
| 速率限制 | IP 60/min + 设备 30/min |
| 日志脱敏 | cardKey/password/secret 强制 *** |
| 备份加密 | gpg AES-256(ADR 0072) |

**防爆破组合(ADR 0022):**

- IP 限流:60/min
- 设备指纹限流:30/min
- 失败次数锁定:5 次失败锁 1 小时
- 错误响应延迟:0.5s -> 1s -> 2s 指数退避
- 统一错误码 `CARD_INVALID`(不区分格式错/已激活,防时序攻击)

### 3.3 第 3 层:数据安全

**卡密安全(ADR 0014):**

- 格式:4x4 字母数字 + Luhn mod32 校验位(去 0/O/1/I)
- 服务端只存 `SHA-256(cardKey + perCardSalt)`,**不存明文**
- 明文仅生成时返回(一次性)
- 日志只记 `hashCardKey(cardKey, 'log')`(固定 salt,聚合分析)

**多租户隔离(ADR 0018):**

- 所有业务表含 `developer_id` 列
- PostgreSQL RLS:`current_setting('app.tenant_id') = developer_id`
- `TenantPrismaService.tx(developerId, fn)` 在事务内 `SET LOCAL app.tenant_id`
- SDK 入口(handshake)无 JWT,临时 `SET LOCAL ROLE <db_user>`(BYPASSRLS)

**离线缓存加密(ADR 0026):**

- AES-256-GCM
- 密钥派生:`AES_KEY = HKDF(cacheKey + deviceFingerprint, info="offline-cache")`
- 派生在 Rust so 内(不在 Kotlin)
- 缓存带 HMAC 签名,篡改后下次验证直接拒绝
- 存 `/data/data/<pkg>/files/xcj_cache.bin`,权限 0700

### 3.4 第 4 层:客户端核心

**Rust 核心模块(ADR 0023/0071):**

| 模块 | 功能 | 状态 |
|---|---|---|
| `machine_id` | 多因素组合 SHA-256 机器码 | ✅ |
| `crypto` | RSA + AES-256-GCM + HMAC-SHA256 | ✅ |
| `card_key` | Luhn mod32 校验 | ✅ |
| `cache` | 离线缓存加解密 | ✅ |
| `integrity` | APK 签名 hash 比对(简化版,见 ADR 0070) | ✅ |
| `jni_bridge` | JNI 入口(语义化命名 + 可选 opaque-jni) | ✅ |

**可选反逆向 features(默认关,开发者按需启用):**

| feature | 效果 | ADR |
|---|---|---|
| `obfstr` | 字符串混淆(编译时加密 URL/密钥/错误码) | 0061 |
| `opaque-jni` | 额外导出 native01-08 非语义化别名 | 0071 |

**撤除的设施(见 ADR 0069/0071):**

- ❌ .text 段自校验(开源 + 服务端验证权威,ROI 低)
- ❌ 反调试 / VM 检测(ADR 0024 未实现,待 P2)
- ❌ 控制流平坦化(ADR 0061 未来工作,SaaS 独占)

### 3.5 第 5 层:完整性校验

**APK 签名 hash 比对(ADR 0070 简化实现):**

- 客户端 `integrity.rs` 计算整文件 SHA-256
- 比对白名单(`verify_apk_signature(apk_path, allow_list)`)
- 大小写不敏感
- **服务端白名单下发接口待 P2.2 实现**

## 4. 威胁模型

### 4.1 威胁等级分类

| 等级 | 威胁 | 防护层 | 残余风险 |
|---|---|---|---|
| 高 | 卡密暴力枚举 | 第 2 层限流 + 失败锁定 | 攻击者换 IP/设备仍可慢速枚举 |
| 高 | 客户端 so 被 patch | 第 4/5 层 + 服务端验证 | 缓存期内可绕过(7 天) |
| 中 | 跨租户数据泄露 | 第 3 层 RLS | RLS 策略 bug 导致泄露 |
| 中 | 中间人攻击 | 第 1 层 HTTPS + 应用层加密 | 证书伪造(依赖 CA 信任) |
| 中 | 卡密明文日志泄露 | 第 2 层日志脱敏 | 拦截器 bug 导致明文输出 |
| 低 | 逆向 SDK 算法 | 第 4 层 Rust so | 开源下不可避免 |
| 低 | 离线缓存被 patch | 第 3 层 HMAC 防篡改 | 缓存期内可绕过 |

### 4.2 攻击场景与响应

**场景 1:攻击者枚举卡密**

```
攻击:同 IP 1 分钟发 1000 次激活请求
防护:IP 限流 60/min,超过返回 429
响应:第 20 次失败触发告警(飞书机器人),第 5 次失败锁 IP 1 小时
```

**场景 2:攻击者 patch 客户端跳过验证**

```
攻击:改 Kotlin 层 if (result.success) 永远 true
防护:服务端验证是权威,patch 客户端不影响最终验证
局限:攻击者需联网验证时仍被拒;但离线缓存期内(7 天)可绕过
响应:缓存过期后必须联网,服务端可下发"强制下线"指令(ADR 0017,待实现)
```

**场景 3:攻击者重打包 SDK 注入恶意逻辑**

```
攻击:改 SDK 后重打包分发
防护:APK 签名 hash 白名单(第 5 层)
局限:白名单需服务端下发(P2.2 待实现),当前白名单为空时跳过校验
响应:服务端下发白名单后,签名不匹配的 APK 拒绝验证
```

**场景 4:开发者后台被入侵**

```
攻击:盗取开发者 JWT,访问后台
防护:JWT 15min 过期 + refresh 7day 可撤销 + 2FA
响应:开发者可撤销所有 refresh token,审计日志记录所有操作
```

## 5. 已知限制

我们坦诚公开以下限制(详见 ADR 0042 公开边界):

### 5.1 客户端限制

- **客户端逆向不可避免**:开源下 Rust so 会被逆向,我们只提高成本不追求绝对防护
- **客户端 patch 不可避免**:攻击者可 patch 客户端跳过校验,服务端验证是权威
- **离线缓存期内无服务端校验**:默认 7 天缓存期内,patch 客户端可绕过验证
- **反调试 / VM 检测未实现**:ADR 0024 待 P2 实现,模拟器批量刷卡密无客户端门槛

### 5.2 服务端限制

- **单机部署**:MVP 单机,无异地容灾(ADR 0072 备份方案)
- **无 WAL 流复制**:RPO 24h,凌晨故障丢前一天数据
- **无 KMS 密钥管理**:备份密钥本地 `/etc/xcj-backup.key`(权限 600)
- **无风控引擎**:ADR 0022 标记"风控引擎放 v2",当前用规则

### 5.3 运维限制

- **单人开发**:无安全团队,响应时间 48h(非 24h)
- **无渗透测试**:MVP 阶段无专业渗透测试,依赖社区披露
- **无 Bug Bounty**:无漏洞赏金,依赖负责任披露

## 7. 安全审计

### 7.1 自审计

- 每次 milestone 结束对照 ADR 0019-0027 安全决策 review
- CI 跑 `npm audit` + `cargo audit`(ADR 0034,待补 Trivy)
- 后端覆盖率 ≥ 80%(当前 84.19%,CI 卡点)
- Rust 核心覆盖率 ≥ 90%(ADR 0038)

### 7.2 社区审计

- 开源后接受社区 PR review(核心 Rust 模块除外,ADR 0004)
- 安全漏洞走 [SECURITY.md](../SECURITY.md) 披露流程
- 修复后公开披露细节 + CVE
